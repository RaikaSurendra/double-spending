THE DOUBLE-SPENDING PROBLEM

Two servers. One Redis. A race condition that silently corrupts your balances — and why a Mutex won't save you.

01 · The Problem

A race condition you can't see — until the money is gone

When two servers read the same balance from Redis, compute locally, and write back — the second write silently overwrites the first. Funds vanish into the gap between read and write.

Both servers read the same balance
Server A calls GET balance → receives 100. A millisecond later Server B calls GET balance → also receives 100. Both now hold the same stale value.
T = 1ms
Each server computes locally
Server A: 100 − 10 = 90. Server B: 100 − 10 = 90. The subtraction happens in application memory — completely outside Redis.
T = 2ms
Server A writes back → Redis = 90
Server A calls SET balance 90. Correct so far — one deduction applied.
T = 3ms
Server B overwrites → Redis = 90 again
Server B calls SET balance 90. It computed from the stale value of 100, not the updated 90. Server A's write is gone. 10 units have been double-spent.
02 · The Wrong Fix

A Mutex only locks your own memory

Adding sync.Mutex enforces order inside one process. But Server B has its own mutex in its own memory space — it cannot see yours.

🚩

Red flag in interviews. "I'll wrap the logic in a Mutex" signals you're thinking about in-process synchronization, not distributed state. A mutex is a memory primitive. Redis is external memory. Your lock has zero authority over it.

03 · The Right Fix

Push the arithmetic into Redis

Redis executes commands on a single main thread. DECRBY is one atomic instruction — there is no gap between the read and the write. Nothing can interleave.

⚡

The key insight: Instead of reading a value, computing locally, and writing back (three steps, three opportunities for races) — collapse all three into a single atomic Redis command. No application-level arithmetic required.

04 · Advanced Fix

Lua scripts: conditional atomicity

When you need to check the balance before deducting — preventing overdraft — a single DECRBY isn't enough. A Lua script via EVAL runs entirely inside Redis, atomically, with full conditional logic.

Why Lua is atomic: Redis guarantees no other command executes while a Lua script is running. The GET and DECRBY inside the script form a single, uninterruptible unit — even across hundreds of concurrent clients.

05 · Key Principles

Staff-level thinking on distributed money

Four rules to internalize before touching a balance in a distributed system.

⚡

Atomic Redis instructions

Use INCR, DECR, INCRBY, DECRBY for simple arithmetic. Redis's single-threaded execution model makes these inherently race-free.

📜

Lua for complex logic

When you need conditionals (balance ≥ amount before deducting), use EVAL with a Lua script. The entire script is atomic — check and deduct are one operation.

🚫

Never cache-only for money

Redis is fast but can lose data on crash. Never use an in-memory cache as the sole source of truth for financial balances. Always back it with a durable store.

🔑

Durable ledger + idempotency

Every deduction should be an immutable ledger entry. Pair each operation with an idempotency key so retries are safe and duplicate charges are impossible.

cmd/broken/main.go RACE CONDITION 
// Non-atomic read-modify-write — triggers race condition
func main() {
    rdb := redis.NewClient(&redis.Options{Addr: "localhost:6379"})
    ctx := context.Background()
    rdb.Set(ctx, "balance", 2000, 0)

    var wg sync.WaitGroup
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()

            // Step 1: Read current balance
            val, _ := rdb.Get(ctx, "balance").Int()

            // Step 2: Simulate processing time — widens the race window.
            // Many goroutines read the same stale value before any write.
            time.Sleep(time.Millisecond)

            // Step 3: Write locally-computed value back.
            // Other goroutines have ALREADY overwritten this key.
            rdb.Set(ctx, "balance", val-10, 0)
        }()
    }
    wg.Wait()

    final, _ := rdb.Get(ctx, "balance").Int()
    // final will be MUCH higher than expected 1000
    // e.g.: Expected: 1000  Actual: 1950  Lost: 950
}
cmd/mutex-wrong/main.go STILL BROKEN 
// Each server has its own mutex — they cannot see each other's lock.
// Server B's mutex does NOT prevent it from reading a stale value
// that Server A has already updated.

func runServer(ctx context.Context, rdb *redis.Client, n int, mu *sync.Mutex, wg *sync.WaitGroup) {
    for i := 0; i < n; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()

            // Locks THIS server's mutex only.
            // The other server has a DIFFERENT mutex in a DIFFERENT process.
            mu.Lock()
            defer mu.Unlock()

            val, _ := rdb.Get(ctx, "balance").Int()
            time.Sleep(time.Millisecond) // race window still wide open
            rdb.Set(ctx, "balance", val-10, 0)
        }()
    }
}

func main() {
    rdb := redis.NewClient(&redis.Options{Addr: "localhost:6379"})
    ctx := context.Background()
    rdb.Set(ctx, "balance", 2000, 0)

    var mutexA, mutexB sync.Mutex // TWO separate mutexes — invisible to each other
    var wg sync.WaitGroup

    runServer(ctx, rdb, 50, &mutexA, &wg) // "Server A"
    runServer(ctx, rdb, 50, &mutexB, &wg) // "Server B"
    wg.Wait()

    // Still corrupted: Expected: 1000  Actual: ~1490
}
cmd/fixed-incrby/main.go CORRECT 
// DECRBY is a single atomic Redis instruction.
// There is NO gap between read and write — Redis does both internally.
// No goroutine, no server, no race can interleave.

func main() {
    rdb := redis.NewClient(&redis.Options{Addr: "localhost:6379"})
    ctx := context.Background()
    rdb.Set(ctx, "balance", 2000, 0)

    var wg sync.WaitGroup
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            // One atomic instruction — no read, no compute, no write.
            // Redis handles all three internally, on its single main thread.
            rdb.DecrBy(ctx, "balance", 10)
        }()
    }
    wg.Wait()

    final, _ := rdb.Get(ctx, "balance").Int()
    // final = 1000 every single time. No exceptions.
    fmt.Printf("Actual: %d  Expected: 1000  Correct: %v\n", final, final == 1000)
}
cmd/fixed-lua/main.go CORRECT + OVERDRAFT PROTECTION 
// Lua script executes atomically inside Redis.
// The balance check AND the deduction are a single uninterruptible unit.
// No other command runs between them — no race, no overdraft.

var deductScript = redis.NewScript(`
local balance = tonumber(redis.call('GET', KEYS[1]))
local amount  = tonumber(ARGV[1])
if balance < amount then
    return redis.error_reply('INSUFFICIENT_FUNDS')
end
return redis.call('DECRBY', KEYS[1], amount)
`)

func main() {
    rdb := redis.NewClient(&redis.Options{Addr: "localhost:6379"})
    ctx := context.Background()
    rdb.Set(ctx, "balance", 500, 0) // Only 500 available

    var wg sync.WaitGroup
    var succeeded, insufficient atomic.Int64

    for i := 0; i < 100; i++ { // 100 goroutines × 10 = 1000 attempted
        wg.Add(1)
        go func() {
            defer wg.Done()
            err := deductScript.Run(ctx, rdb, []string{"balance"}, 10).Err()
            if err != nil {
                if strings.Contains(err.Error(), "INSUFFICIENT_FUNDS") {
                    insufficient.Add(1)
                }
            } else {
                succeeded.Add(1)
            }
        }()
    }
    wg.Wait()

    final, _ := rdb.Get(ctx, "balance").Int()
    // Succeeded: 50, Insufficient: 50, Final: 0 — never negative
}
